# H2 — the `take_value` heuristic (Spender AI variant "H2")

H2 is a from-scratch, 1-ply greedy heuristic built on a single per-card scalar, **`take_value`**.
It is a sibling of the stable variant **H** (`heuristic.py` + `valuation.py`); H2 lives in
`heuristic2.py` + `valuation2.py` and is served on the website as variant **"H2"** (with a 4-value
on-card transparency overlay). 2-player only, pure Python, no ML deps.

Design goal (from the user): rely on **logical variables, not tuned gimmicks**, and always reason
about the **post-cost-reduction** gem cost (cost minus the bonuses from cards you own), never base
cost.

## The model

Every visible board card + the bot's own reserved cards get one scalar:

```
take_value = (engine_value + point_value) / (1 + total_cost)

total_cost  = W_TEMPO*tempo + W_GEM*gem + W_GOLD*gold          # importance tempo > gem > gold
point_value = stage * (PTS + NOBLE_SCALE*noble_progress) + noble_completion   # stage in [floor, 1]
engine_value = valuation2.engine_value                         # undiscounted
```

**Cost terms** — all post-bonus; `tempo`/`gold` are on REMAINING need (also minus held colored
tokens), `gem` is the post-bonus sticker price. **No take-2 is assumed** (you gain ≤1 gem of a
color per turn), so a single color is the bottleneck:
- `tempo` = steepest single-color REMAINING need, `+1` if the remaining cost is exactly `1-1-1-1`
  (four colors → a take-3 + take-1 = 2 turns). The dominant cost term.
- `gem` = total post-bonus gems to pay (the sticker price; held tokens NOT subtracted).
- `gold` = estimated gold coins needed = `max(0, steepest − min(GOLD_BANK_CAP, bank[bottleneck]))`
  (the bottleneck need minus the few of it you can pull from the bank, floored at 0 so a cheap
  bottleneck gives no negative credit — otherwise a 4-gem spread card could score cheaper than a
  3-gem concentrated one).

**Benefit terms:**
- `engine_value` (reused from `valuation2`): the value of the permanent +1 bonus this card grants —
  the discount it gives every *other* card that still needs that color (board + own reserved),
  weighted by each card's worth (`PTS/ENG_DIV + ENG_FLOOR`) and that card's color-hunger, plus a
  forward-looking deck-demand term (`× ENG_DECK_W`). It self-decays as you stack a color (only
  counts cards that *still* need it). **Undiscounted** — a bonus is realized and starts compounding
  the moment you buy the card.
- `point_value` = **game-staged future points + immediate noble VP**:
  `stage·(PTS + NOBLE_SCALE·noble_progress) + noble_completion`.
  - **stage** = `STAGE_FLOOR + (1−STAGE_FLOOR)·min(1, max(cards_bought/STAGE_K, leader_pts/15))` — a
    *global* game-phase multiplier (low early → 1 late). It scales the *future* points so the bot
    builds its engine first and values points as the game develops (early-engine → late-points).
    Being global (same factor for every card) it preserves point proportionality — a 2-pt card is
    exactly 2× a 1-pt card at the same stage.
  - **noble_completion** (the +3 a buy claims *now*) is realized immediately, so it is **NOT staged**.
  - `noble_progress` carries a **closeness floor** (`NOBLE_CLOSE_FLOOR`): a card whose color a visible
    noble needs scores > 0 even at zero bonuses, so noble-relevance survives distance.
  - This replaced an earlier per-card tempo-discount on points. `tempo` is collection *effort* (a
    cost), so it now lives **only** in `total_cost` — it never devalues a card's points or engine.
    The stage swap took H2 from ~0.485 to ~0.62 vs H.

The `+1` in `1 + total_cost` keeps a free card (total_cost 0) finite.

## Tuned config (offline search)

The weights were found by offline search (win rate vs H), then folded in as the committed defaults:

| `heuristic2.py` | | `valuation2.py` | |
|---|---|---|---|
| `W_TEMPO` | 0.5 | `GOLD_BANK_CAP` | 2 |
| `W_GEM` | 0.2 | `ENG_DIV` | 8.0 |
| `W_GOLD` | 0.4 | `ENG_FLOOR` | 0.2 |
| `NOBLE_SCALE` | 3.0 | `ENG_DECK_W` | 3.5 |
| `STAGE_K` | 8 | `NOBLE_CLOSE_FLOOR` | 0.2 |
| `STAGE_FLOOR` | 0.25 | | |
| `ENG_DECAY` | 0.3 | | |

Result: **beats H ~0.68** (wins 69%, draws 0.4%, losses 31% over 1500 games; up from ~0.485 under
the old per-card tempo-discount and ~0.638 before the benefit-side retune). Both play greedy (1-ply,
no MCTS) — the same way they're served on the site.

**Tuning history (benefit-side retune, after the gold floor fix):** with gold floored correctly, the
cost weights (`W_TEMPO`/`W_GEM`/`W_GOLD`/`GOLD_BANK_CAP`) are all saturated — already at their optimum
(CRN sweeps vs H, N=800). The remaining lever was the *benefit* side: H2 was under-valuing the two
dimensions that are H's signature strengths. **`ENG_DECK_W` 1.0→3.5** (forward-looking engine value —
a bonus color's worth across the undealt deck, the biggest single gain) and **`NOBLE_SCALE` 2.0→3.0**
together moved H2 from a lean point-rush toward more engine breadth + nobles, +0.06 win rate. The gain
is additive between the two and holds across three independent seed sets (not CRN-seed-overfit).
Stacking the *redundant* knobs (`ENG_FLOOR`, `NOBLE_CLOSE_FLOOR` — each scales the same axis) on top
over-shot and gave the gain back, so they stay put.

**`ENG_DECAY` 0.0→0.3 (engine horizon-decay):** the conceptual mirror of point staging — engine value
should fall as cards accumulate (a late-built engine has few remaining buys to pay off), since the
deck term of `engine_value` is otherwise flat all game. Tuned carefully: 0.3 is positive on all 6
*fresh* holdout seed sets (11/12 across both rounds); higher over-corrects (0.5 had a real −0.025
downside seed, 1.0 was −0.045), lower (0.1) is a no-op. **But the effect is MINIMAL (~+0.011)** — the
big lever was `ENG_DECK_W`+`NOBLE_SCALE` (~+0.06), not this. Kept for correctness (it's the right
shape), with a likely future cleanup: it reuses `own = cards/STAGE_K`, the same game-progress signal
behind the point `stage`, so the two could be folded into one shared signal (points up, engine down)
instead of a separate `ENG_DECAY` knob. Methodology note: the partial-decay gain *shrank* from ~+0.016
on the tuning seeds to ~+0.006–0.011 on fresh holdout seeds — a reminder that the final number must
come from seeds not used to pick the value (CRN is for the *comparison*; a fresh holdout is for the
*estimate*).

**Take-2-same — tested, rejected (default OFF, `USE_TAKE2`).** The model assumes ≤1 gem/color/turn,
so the bot never takes 2 of one color. Three variants were measured vs H and none earned a place:
- *naive* (take 2 of any unique-steepest color): made the bad move — taking 2 of one color while
  another was also needed — and cost **−0.03**;
- *single-remaining-color* (only one color left to need): fixes the bad move but is still **−0.006**
  (the half-turn saving loses to the option value of the 2 flexible gems a take-3 grabs);
- *reserved-only* (finish a single-color-bottleneck card you've RESERVED): the conceptually-right
  form (a reserved card is locked, so no option-value cost), but ~**neutral (−0.003)** because it
  almost never fires (0.27%): H2's only reserves are *winning* reserves, which are gold-necessary
  (blocked by a gem the bank can't supply) — the opposite of take-2's "bank full of it". To make
  it bite, the bot would need to *speculatively* reserve deep single-color cards then take-2 to
  finish them (untested combined experiment). The reserved-only code is kept behind `USE_TAKE2`.

## Decision flow (`choose_action`)

1. **Winning buy** (secure): an affordable buy reaching 15 (incl. the noble it claims) — taken only
   if *secure* (as first player, reaching 15 gives the opponent a final turn; take it only if they
   can't overtake). If insecure, deny their overtaking card; else take the win.
2. **Winning-via-reserve** (the only proactive reserve on by default): a card that would WIN but is
   unaffordable now *only because a needed gem isn't in the bank* — reserve it to bank the gold and
   win next turn (and lock it). A free win; costs nothing on turns it doesn't trigger.
3. **Endgame denial**: if the opponent can win next turn off a board card, deny it (reserve, else buy).
4. **Token-cap anti-hoard**: near the 10-token cap, buy the best affordable card instead of taking
   (8 tokens → buy if its take_value > `CAP8_BUY_ABOVE` 0.8; 9 → > `CAP9_BUY_ABOVE` 0.5; 10 → always).
5. **Speculative reserve** — OFF (`USE_SPECULATIVE_RESERVE`). See below.
6. **Buy / take**: if the highest-`take_value` card is affordable, buy it; else take gems toward it
   (≤3 distinct colors it needs, spare picks to the next-best). A small `GOLD_TIEBREAK` prefers buys
   that spend less gold among near-ties.

Far cards self-deprioritize via their large `total_cost`, so no reachability/"mirage" gate is needed.

## Reserving — what's in and why

**Reserving is the only source of gold in Splendor**, so it matters for cards blocked by a gem the
bank can't supply. But it costs a full turn, and exhaustive testing showed proactive reserving is a
**tempo drag** for this class of bot:

- **Contested/denial reserving** (reserve what the opponent is ~1 turn from) **over-fired** (~8
  reserves/game) and tanked play (0.48 → 0.20 vs H). Removed.
- **Acquisition** (reserve a card with a big take_value gap) and **general gold-necessary** reserving
  were neutral-to-negative vs H (monotonically worse with frequency). Kept behind
  `USE_SPECULATIVE_RESERVE` (default **False**) for A/B only.
- Measured directly: **H itself gains ≈0 from reserving** (reserve ON−OFF = +0.001); H2's is −0.07.
  So H's elaborate reserve logic is essentially vestigial — H is strong from its *other* machinery,
  not from reserving. (Matches the project's documented "denial is a wash / over-reserving is the
  weakness.")
- **The one unambiguously-correct reserve** is the **winning-via-reserve** (#2 above): a winning
  card blocked solely by an out-of-bank gem is a guaranteed-win setup. It's on, verified end-to-end
  (reserve → spend the gold → win), and rare enough to be aggregate-neutral.

Gold is never *wasted*: the engine pays each color with colored tokens first and only covers the
unavoidable shortfall with gold (`engine.py` buy routine), and `_choose_discard` dumps gold last.

## Serving + overlay

- `main.py`: `_ai_variant_valid` accepts `"H2"`; `_h2_choose_move` runs `heuristic2.choose_action`
  in the thread pool (same path as H/Z); `_h2_card_values` produces the per-card overlay.
- The overlay shows four values per card — **T**(ake) / **E**(ngine) / **P**(oint) / **C**(ost) —
  computed from a single shared `heuristic2.components(...)` so the bot and the overlay never drift.
- Frontend (`Spender.jsx`): "H2" in the lobby "Play vs AI" picker; the 4-value box renders when the
  overlay value is the H2 object (H's single-number overlay is unchanged).

## Notable decisions (do not relitigate)

- The point **distance-discount** (`RATE`) is **gone**. It conflated collection *effort* (per-card
  distance — which belongs in `total_cost`) with a card's intrinsic worth, and it distorted point
  proportionality (a 2-pt card didn't read as 2× a 1-pt card). It was replaced by the global **stage**
  multiplier above — cleaner *and* stronger (~0.485 → ~0.62 vs H).
- The literal sanity check "every L1 0-pt > every other card on turn 1" is **infeasible** (a
  high-demand-color L2 legitimately beats a junky L1, ~40/60 boards). The meaningful version — "the
  top take_value card on turn 1 is an L1 0-pt card" — is what the early stage floor produces.
- 0.70 vs a **strong** (deep-search) C2 is unreachable by any heuristic (H also drops to ~0.42); that
  bar needs the AZ search stack. H2's "~0.81 vs C2" is vs *greedy* C2, the budget H's strength is
  quoted at.
